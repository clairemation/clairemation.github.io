The Big O

What, you thought I was above that joke?

Anyway, Big O notation is a very general way to describe how efficient an algorithm is. In other words, as you feed your algorithm huger and huger sets of data, does it run a little bit slower, or a lot slower? And how much more data does it take to make it start choking? Big O expresses the relationship between the data set and the speed of the programâ€”as the data set increases, how quickly does the runtime increase?

O(1)

Take this function:

function doNothing(bunchOData) { return; }

It does just what it says on the box. Nothing. And that's what it does, and will always do, no matter how big or small bunchOData is. In other words, it's constant. So we call its efficiency O(1). If you plotted the relationship on a graph, it would be a flat line.

O(n)

This describes an algorithm where the runtime increases in direct proportion to the data set, whose size we call n. On a graph it would be a straight diagonal line. A for or each loop would give you O(n), because it runs one additional time for each additional piece of data.

O(n^2)

So if one loop is n, what are two nested loops? n * n, or n^2. Still a linear relationship, but a steeper one.

O(2^n)

Uh-oh. Exponents. This notation indicates that each addition to the data set will double the amount of time it takes to crunch. So, it grows exponentially. If you remember your TI-84 calculators (and remember doing anything on them other than playing Drug Wars), this should give you images of an upward curve that rapidly gets steeper and steeper until it approaches (but never reaches) sheer vertical. An asymptote. Recursive algorithms can take O(2^n) time.
